# Documentação funcional
Documentação funcional, também conhecida como documentação de produto ou de visão geral, é uma documentação que tem o objetivo de contextualizar e apresentar um assunto. Este assunto pode ser um produto, um serviço, uma funcionalidade, um recurso, enfim, apresentar uma temática para a pessoa usuária.

Geralmente, são aquelas documentações que abrem os portais de documentação dos produtos, pois é por elas que devemos começar para nos situar sobre o que é determinado produto, o que é determinado serviço, como ele funciona.

<br/>

## Tópicos para estruturar sua documentação
Falando um pouco sobre a estrutura desse tipo de documentação, quais são os tópicos que geralmente compõem uma documentação funcional?

Começamos com **o que é o produto**. Não precisa ser exatamente esse título na sua documentação, mas normalmente é uma seção, um tópico dentro do documento cujo objetivo é responder essa pergunta. Então, às vezes, pode ser visão geral ou overview.

Também temos **um breve contexto do seu uso**. Na seção de visão geral, além dessa seção, podemos deixar uma lista com tópicos que abordaremos na sequência, e depois abordar cada um desses tópicos.

Na sequência, temos os **recursos disponíveis**. Por exemplo: se falamos da documentação de um banco, uma documentação funcional ou de visão geral de um banco pode ter o nome do banco e uma breve apresentação dele, os serviços que ele oferece, e uma descrição detalhada de cada um dos serviços.

Além disso, temos **para que e para quem serve** esse produto ou serviço, porque, assim, direcionamos quem é o público dessa documentação.

Também temos os **principais conceitos**. Às vezes, essa seção de principais conceitos, que também pode ser um glossário, não precisa estar exatamente no mesmo arquivo em que escrevemos essa documentação funcional. Ela pode ser uma seção à parte, mas com um hiperlink, uma referência para que a pessoa acesse essa documentação.

Analisando juntos os tópicos, já conseguimos entender que essa é realmente uma documentação que introduz, contextualiza, apresenta, e dá um primeiro panorama para a pessoa usuária sobre determinado produto ou serviço.

<br/>

## Analisando exemplos
Porém, para tornar isso ainda mais tangível, vamos analisar um exemplo de documentação. Para facilitar, decidimos usar o próprio exemplo do GitHub, especificamente de uma documentação chamada "**[Introdução à sua conta do GitHub](https://docs.github.com/pt/get-started/onboarding/getting-started-with-your-github-account)**".

Apenas pelo título, percebemos que ele vai apresentar o recurso de como é criar uma conta ou ter uma conta no GitHub. Ele inicia com a seguinte descrição: "*Este guia descreverá a configuração da sua conta do GitHub e uma introdução aos recursos de colaboração e comunidade do GitHub*".

Já fica muito evidente que é uma documentação funcional, porque, ainda que tenha partes, cada uma delas é sobre uma contextualização geral de como funciona uma conta no GitHub.

Temos seções como "Personalizando e automatizando seu trabalho em GitHub", "Criando com segurança em GitHub", "Usando ferramentas e processos de GitHub". São vários recursos que a conta do GitHub oferece e essa documentação vai apresentar como funciona cada um deles.

Se rolarmos a tela, percebemos que é uma documentação extensa, porque como ela tem o objetivo de apresentar e contextualizar sobre esse cenário, acaba sendo extensa de forma proposital, para dar o máximo de contexto possível, de modo que, ao final dessa leitura, você se sinta confortável para começar a fazer cada vez mais integrações e ações com aquele produto.

Porém, como faremos no nosso projeto? Começando a mostrar as partes práticas do projeto, teremos um modelo de documentação funcional no nosso repositório. Como nosso caso de uso é o banco ByteBank, decidimos criar simulando como seria uma [documentação para apresentar o ByteBank](https://github.com/marimoreiratw/projeto-alura/blob/main/documenta%C3%A7%C3%A3o-funcional.md).

Temos o título "ByteBank", seguido de uma breve descrição ("*A ByteBank é um banco digital que foi criado com objetivo de simplificar a vida financeira dos brasileiros*"), e em seguida começamos:

```
A seguir, você encontra mais informações sobre:

- Serviços disponíveis
- Para quem é a ByteBank?
- Como criar sua conta
```

O recurso acima é muito interessante, e na nossa área é chamado de sumário (ou table of contents), que é exatamente quando, ao início de uma documentação, trazemos todos os itens e assuntos que ela vai abordar, já antecipamos para a pessoa usuária o que ela vai encontrar naquele conteúdo. É um recurso muito interessante, que vale a pena e que é muito valioso em uma documentação funcional.

No nosso caso, como estamos simulando um contexto de apresentar como funciona a ByteBank, os tópicos que colocamos foram "Serviços disponíveis", "Para quem é a ByteBank?", e "Como criar sua conta", que seria o primeiro passo para começar a se relacionar com a ByteBank.
